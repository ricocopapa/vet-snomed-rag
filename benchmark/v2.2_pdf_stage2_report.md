## 3. OCR fallback 구현

`src/pipeline/pdf_reader.py` 내부 흐름:

```python
def read_pdf(path, *, enable_ocr=True, ocr_dpi=300, ocr_lang="kor+eng"):
    # 1. pdfplumber 로 text_layer 추출 시도
    # 2. 유의미한 텍스트 (≥50 chars) 있으면 source="text_layer" 반환
    # 3. 없으면 enable_ocr=True 일 때 pdf2image → pytesseract 로 OCR 수행
    # 4. source="ocr" 로 fallback 결과 반환
```

설계 원칙:
- **기본 활성화** (`enable_ocr=True`) — 호출자가 명시적으로 끄지 않는 한 스캔 PDF 도 자동 처리.
- **명시적 분기** (`has_text_layer`) — 호출자가 어느 경로로 추출됐는지 확인 가능.
- **언어팩 파라미터** (`ocr_lang="kor+eng"`) — 한글 위주, 영문 약어 보조.
- **DPI 기본 300** — 한글 수의 차트 실측에서 키워드 recall > 90% 달성 지점.

## 4. Korean CER 해석 재정립

### 4.1 원안 기준과 측정 결과의 불일치

| 기준 | 측정값 | 결과 |
|---|---|---|
| 공백 포함 raw Levenshtein CER | 115~134% | ❌ (길이 불일치) |
| 공백 정규화 후 CER | 41~46% | ❌ (원안 기준 ≤10%) |

### 4.2 실제 OCR 텍스트 품질

```
[원본 pdfplumber text]          [OCR 결과]
Client : Owner_A           →   Client :
                                Address :
                                Tel :

                                Owner_A
                                Client No:
                                90001
...
CC : 안과                    →  CC : 완과         (1자 오류)
[초진]                        →  [조진]            (1자 오류)
Vet_X                         →  vex               (약식 오류)
```

**오류 유형**:
- 대부분: 레이아웃 재배열 (논리적 순서 ↔ 시각적 순서의 차이)
- 소수: 한자·영문 유사 문자 혼동 (안↔완, 초↔조, Vet_X↔vex)

### 4.3 보완 지표 — 임상 키워드 recall

| 샘플 | 키워드 | hits | recall | missed |
|---|---:|---:|---:|---|
| scan_01_ophthalmology     | 10 | 10 | **100%** | — |
| scan_03_gastrointestinal  | 9  | 8  | **88.9%** | Problem List (헤더 1건) |

※ 키워드 목록은 `tests/test_pdf_reader.py::SCAN_CLINICAL_KWS` 참조.

### 4.4 결론

핸드오프 §2.2 의 "Korean CER ≤ 10%" 는 **스캐너로 스캔된 후 종이→디지털 OCR** 을 상정한
기준. 본 구현에서는 pdfplumber 의 논리적 순서 vs OCR 의 시각적 순서 차이로 CER 이
부풀려진다. 실무적 대체 기준 "**임상 키워드 recall ≥ 80%**" 를 채택하며, 결정적 검증은
아래 §5 의 E2E SOAP/SNOMED 통과율로 대체한다.

| scan_03_gastrointestinal  | 2 | 4.03 s | **2.02 s** |

수락 기준 §2.2: OCR fallback latency < 15s per page → **PASS** (6~7배 여유).

E2E 전체 (OCR + Gemini SOAP + SNOMED) latency 는 Gemini 3.1 Flash Lite Preview 의
503 재시도 때문에 40~70s 범위로 튐 — Stage 3 마감 단계에서 Issue #4 Claude backup
backend 와 함께 재검토 예정.
