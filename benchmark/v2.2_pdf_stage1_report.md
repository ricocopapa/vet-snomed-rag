## 3. `pdf_reader.read_pdf()` 반환 스키마

```python
{
    "text": str,             # 전체 페이지 텍스트 (\n 으로 연결)
    "pages": int,            # 페이지 수
    "has_text_layer": bool,  # 유의미한 텍스트 레이어 존재 여부
    "source": str,           # "text_layer" | "ocr" | "vision"
}
```

판정 임계값: `TEXT_LAYER_MIN_CHARS = 50`. 향남 샘플 최소 chars=870 이므로 확실히 통과,
빈 스캔 PDF 는 `has_text_layer=False` + `source="ocr"` 로 Stage 2 훅에 전달될 예정.

## 4. Latency 측정

`.venv/bin/python` 환경에서 warm-up 3 라운드 후 5 라운드 × 3 샘플 = 15 측정.

| 샘플 | n | p50 (ms) | p95 (ms) | max (ms) |
|---|---:|---:|---:|---:|
| hyangnam_anon_01_ophthalmology.pdf      | 5 | 49.05 | 49.32 | 49.32 |
| hyangnam_anon_02_dermatology.pdf        | 5 | 35.99 | 36.38 | 36.38 |
| hyangnam_anon_03_gastrointestinal.pdf   | 5 | 47.35 | 47.79 | 47.79 |
| **OVERALL**                              | 15 | **47.35** | **49.32** | **49.32** |

수락 기준: `pdf_reader` 단독 p95 < 5s → **PASS (100배 여유)**.

참고 사항: E2E 전체 latency (PDF → SOAP → SNOMED) 는 Gemini API 경로에 의해 좌우되며
본 Stage 범위에서는 측정하지 않는다. v2.1 대비 E2E Precision 회귀 측정은 실 API 호출
권한이 필요하므로 다음 체크포인트 (§8) 에서 수행한다.

## 6. 파이프라인 통합 Smoke Test (dry-run)

`ClinicalEncoder(dry_run=True)` + 3 PDF 입력:

| PDF | 탐지 도메인 | 추출 필드 | SNOMED 매핑 성공 | SOAP 필드 |
|---|---|---:|---:|---|
| 01_ophthalmology       | `['VITAL_SIGNS']` *                       | 4 | 4/4 | O |
| 02_dermatology         | `['VITAL_SIGNS']` *                       | 4 | 4/4 | O |
| 03_gastrointestinal    | `['GASTROINTESTINAL', 'VITAL_SIGNS']`     | 5 | 5/5 | S+O |

(*) dry-run `_mock_response` 는 하드코딩된 키워드 세트 (예: `각막 형광 / 외이염 / 설사`)
로 도메인을 판정한다. 향남 샘플의 실제 텍스트는 `외상성 각막염` (`각막 형광` 아님) /
`외이염 재진` 등으로 일부 키워드만 일치해 01/02 는 `VITAL_SIGNS` 폴백에 빠진다. 이것은
**dry-run mock 의 한계**이지 `pdf_reader` 또는 SOAP 파이프라인 결함이 아니다.
실 Gemini API 호출 시 도메인 탐지는 정상 동작할 것으로 예상한다 (Stage 2 진입 시 재확인).

결함 없음. 예외 0건. 파이프라인은 3/3 PDF 에서 SOAP 필드 추출 + SNOMED 매핑까지 통과.
